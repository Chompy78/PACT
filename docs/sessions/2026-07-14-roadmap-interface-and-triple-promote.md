# 2026-07-14 — Roadmap interface + three-PR promotion to main

## What happened
Built a new standalone visual roadmap page (`docs/roadmap.html`) on request — a self-contained,
vanilla-JS single page with seven views (Board, Timeline, Category, Priority, radial Map, Table,
Dashboard) driven by a shared search/status/category/priority filter bar. Data is a point-in-time
snapshot: open items hand-authored from `docs/PACT_ROADMAP.md`, shipped items parsed from `CHANGELOG.md`.
Then linked it from `index.html` (a new "Project" section with a Roadmap card in the existing tool-card

Separately, at the user's direction ("merge all and make main up to date"), promoted **three** things to
production in one sitting.

## The notable part — a three-PR promotion with a cross-PR conflict
"Merge all" was ambiguous: besides this session's roadmap work, two *other* sessions' PRs were open
against `preview` — **#205** (campaign join/redeem race → friendly error; live-DB SQL/RLS) and **#206**
(advancement dials; touches `js/engine.js` + tools, flagged by its own author as "needs a manual
in-browser pass before shipping to main"). Asked via `AskUserQuestion`; user chose to include all three.

Merged locally (not via sequential GitHub API merges) because #205, #206, and the roadmap work all
prepend to the top of `CHANGELOG.md`/`DECISIONS.md`/`PACT_ROADMAP.md`. The only real conflict was in
`PACT_ROADMAP.md`: #205 and #206 had each *graduated its own task* out of the NEXT section, so the two
sides of the conflict were "keep advancement-tracks, drop race-fix" vs. "keep race-fix, drop
advancement-tracks". Correct resolution was to drop **both** (both were now shipped). Resolved locally in
one pass, pushed `preview`, then promoted `preview` → `main` via PR #207 after full CI went green
(parity + log-fuzz + e2e).

## Why local-merge over API-merge
Sequential API merges would each land clean, then re-conflict the next PR against the new top-of-file —
N round-trips of conflict resolution through the API (which can't resolve conflicts). Integrating all
branches into one local `preview` and resolving the shared-file conflict *once* is strictly less work and
less error-prone. This is the generalizable lesson pushed to `ai-lessons-learned`.
