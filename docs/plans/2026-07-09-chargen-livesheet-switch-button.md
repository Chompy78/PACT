# Plan: One-click "switch tool" between CharGen and Live Sheet, on a shared character-store module

## Goal
Today, moving an in-progress character from CharGen to Live Sheet is a manual dance (export → download/share
a JSON file → open the other tool → import → pick the file), and there is no reverse direction at all. This
plan delivers a single button in each tool ("Open in Live Sheet" / "Open in CharGen") that navigates
straight to the other tool with the same character already loaded — like switching tabs. The two HTML files
stay physically separate (a full one-file merge was considered and explicitly rejected). **The work is
sequenced shared-module-first:** rather than hand-copy the new cross-tool glue into two ~300–500KB files, a
small shared `js/character-store.js` is created first and the switch button is written against it from the
start — the first concrete payoff of the finished "two views over one shared character log" architecture.

## Context
Static, vanilla-JS, no build step, served on GitHub Pages under one origin. Two large independent HTML tools
here: `tools/PACT-CharGen-Webtool.html` (creation) and `tools/PACT-Live-Char-Sheet.html` (play). A prior,
**already-executed** plan (`docs/plans/2026-07-08-chargen-livesheet-unification-phase2.md`, "Phase 2")
rewrote CharGen to share Live Sheet's runtime model — both now represent a character as an append-only event
log (`LOG`/`SEQ`) replayed through shared engine functions. Phase 2's principle: *"the character is the
LOG... two interchangeable UI views over one shared character model,"* and it explicitly left **physical
file merge out of scope** (a decision re-confirmed by the task owner for this plan).

Two repo rules verified in `AGENTS.md`, both of which shape the sequencing here:
- **`js/` holds shared logic; `tools/` are UI-only.** Persistence is today's biggest violation — each tool
  hand-copies its own save/load and an identical `genCharId()`. This plan starts to correct that.
- Rules logic lives only in `js/engine.js`; this plan touches **no** rules logic (no `compute()` change).

Per Phase 2's own implementation notes, `emit()`/`undo()`/`render()` are legitimately **tool-local** (they
manipulate each tool's own `LOG`/`SEQ`/DOM, not rules) and must stay so. That fixes the architectural
boundary for the shared module below: it owns **transport/storage/validation** (tool-agnostic), never a
tool's runtime application of a character (tool-specific).

This plan was shaped by two independent cold reviews plus a deeper code-verified pass; see "Review outcome."

## Assumptions vs. verified facts
- **Verified (read/tested directly in the code this session):**
  - CharGen's `exportToLiveSheet()`/`buildToLiveLog()` is a manual file/share flow — no live handoff exists.
  - Both tools import `foldBuild`/`activeEvents`/`economy`/`MUT` from `js/engine.js` (D-GH33/36/37); a raw
    `LOG` folds identically in either tool, no format conversion needed.
  - `genCharId()` is **byte-identical** in both tools (`'c'+Date.now().toString(36)+Math.random()...`) —
    safe to migrate to a shared module.
  - Each tool's module bridge fires an `engine-ready` event and boot (`_lsBoot`/`_cgBoot`) is gated on it
    (`if(window.compute){boot()}else{addEventListener('engine-ready',boot)}`). A `character-store.js`
    imported in that same bridge and exposed on `window` is therefore **guaranteed available before boot
    runs** — same pattern engine symbols already use.
  - Live Sheet local save: single fixed key `KEY='pactLiveSheet'`, `save()` writes
    `{LOG,SEQ,rules:RULES,id:currentCharId()}`, `load()` reads it. Live Sheet's `render()` does **not**
    autosave; every mutation pairs an explicit `save()` with `render()`.
  - CharGen local autosave: single fixed key `CG_AUTOSAVE_KEY='pactCharGenAutosaveV2'`, envelope
    `{schema:'pact-chargen/1',rules,name,budget,LOG,SEQ,id}`; CharGen's `render()` **does** call
    `_cgAutosave()` at its end unconditionally.
  - **The correct CharGen consume primitive is `_cgApplyEnvelope()`, NOT the untagged "branch 2" import
    path.** `loadFile()` branch 1 (`_cgApplyEnvelope`) sets `b.id=d.id` and reinstates `LOG`/`SEQ` verbatim;
    branch 2 (`foldBuild`+`applyBuild`) **never reads `d.id`** and mints a fresh random id — using it would
    silently break id-stability. (This was an actual bug in an earlier draft, caught by reading the code.)
  - **Tested in a real headless browser:** an autosave-restored character keeps its `id` but its raw `LOG`
    is regenerated by a trailing `_cgBoot()` step (`replaceWholeLogFromBuild(_domReadBuild())`, dead
    Phase-2-Step-3 scaffolding) — event counts shift across boots. This does **not** corrupt the character:
    a Medium-fixed Dwarf round-trips with stored `size:'Small'` but `compute()` derives the true `Medium`
    and all downstream values use compute's, so the character is **compute-equivalent**. Conclusion: the
    switch preserves *the character*, not the *exact event log* — and shouldn't be relied on to.
  - CharGen has **zero** cloud/auth/campaign wiring (no imports from `js/auth.js`/`sync.js`/`campaign.js`).
    Live Sheet has all three.
  - `service-worker.js` serves `.html` network-first (regex on pathname; `?handoff=` doesn't change the
    match). Online: fresh fetch, query intact. Offline: a unique handoff URL is never cached, so the SW
    falls back to `/PACT/index.html` — the switch is silently an **online-only** feature.
  - No existing use of the string `handoff` anywhere in the repo (no key/name collision).
- **Assumed (flagged, not yet confirmed):**
  - A cloud-campaign-bound character can be handed as "just a LOG + id" and reattach on return, i.e. the
    campaign relationship lives in Supabase against the `id`, not inside the LOG. **Not verified** against
    `js/sync.js`/`js/campaign.js`/`sql/schema.sql` — this is why cloud is isolated in Slice 2.

## Architecture: what the shared module owns (and deliberately does not)
`js/character-store.js` (new) owns **tool-agnostic transport, storage, and validation**:
- `genCharId()` — migrated verbatim (the first shared primitive, establishing the pattern).
- `writeHandoff(payload) → uuid` — stamps `{schema:'pact-handoff/1', source, created:Date.now(), LOG, SEQ,
  rules, id}`, writes it under `pact:handoff:<uuid>`, returns the uuid for the caller to put in the URL.
- `takeHandoff(uuid) → payload | null` — reads the one named key, validates (`schema` match, `LOG` is an
  array, `created` within the expiry window), deletes it, returns the payload or null on any failure.
- `sweepExpiredHandoffs()` — deletes **all** `pact:handoff:*` keys past expiry (orphan cleanup).
It does **not** own: `emit`/`undo`/`render` (tool-local), nor the *application* of a consumed payload to a
tool's runtime — that stays in each tool (`_cgApplyEnvelope` for CharGen; direct `LOG/SEQ/__charId`
assignment mirroring `importJSON` for Live Sheet), because those touch tool-local state and differ by tool.
This mirrors exactly how `engine.js` is shared while `emit`/`undo` are not. It also keeps Slice 0 a true
no-behavior-change refactor: the module is new code + one migrated one-liner, nothing existing is rewired.

## Proposed approach — sliced, shared-module-first
**Slice 0 — `js/character-store.js` (groundwork, no user-visible change).**
1. Create the module with the four exports above. `writeHandoff`/`takeHandoff`/`sweepExpiredHandoffs` are
   brand-new code (born shared, never duplicated). `genCharId` moves in verbatim.
2. Both tools import it in their existing `<script type="module">` engine bridge and expose the functions on
   `window` (same mechanism as engine symbols), so the classic scripts can call them. Delete each tool's
   local `genCharId` in favour of the shared one.
3. Verify no behavior change: both tools still boot, save, load, and generate ids exactly as before;
   `engine-parity.html` unaffected.

**Slice 1 — the switch buttons, written against Slice 0 (local characters only).**
4. **Buttons:** add `switchToLiveSheet()` in CharGen (button "Open in Live Sheet", beside the existing
   `exportToLiveSheet()` which stays) and `switchToCharGen()` in Live Sheet ("Open in CharGen"). Each calls
   its own local persistence path first (CharGen `_cgAutosave()`; Live Sheet `save()` — order vs. the
   handoff write is immaterial, both already reflect current state), then
   `const u = writeHandoff({source, LOG, SEQ, rules, id})` and navigates `?handoff=<u>` to the other tool.
5. **Boot precedence (explicit in both boot functions), so exactly one path fires:**
   1. `sweepExpiredHandoffs()` runs first (cleanup, unconditional).
   2. If `?handoff=<uuid>` present and `takeHandoff(uuid)` returns a payload → **consume it, skip both the
      normal local-load and any deep-link check** for this boot. Consume = apply via the tool-local
      primitive (`_cgApplyEnvelope({LOG,SEQ,id,rules},{clearHistory:true})` for CharGen; direct
      `LOG/SEQ/__charId/loadedRules` assignment then `save()` for Live Sheet), then `history.replaceState`
      to strip the query string. Set `loadedRules` from `payload.rules` so each tool's **existing**
      stale-rules banner/reprice indicator fires correctly.
   3. Else existing deep-link (CharGen `loadFromHash()`), only if no handoff fired.
   4. Else normal autosave/last-saved load. 5. Else blank.
   (CharGen already suppresses autosave-restore after a successful `loadFromHash` via `_cgHadShared`; the
   handoff slots ahead of both using the same one-of-N pattern.)
6. **Overwrite safety:** the destination's own autosave slot is updated only *after* a successful parse, so
   a corrupt/expired handoff can never clobber a good pre-existing autosave.

**Slice 2 — cloud/campaign round-trip (isolated; the only part carrying cloud risk).**
7. Using Slice 1 unchanged, verify (or fix) the cloud-bound case per the objective checks in Verification.
   Fallback if it doesn't round-trip cleanly: disable/hide the switch button for campaign-bound characters
   in Live Sheet (CharGen has no campaign concept to lose), and log the decision — this never blocks Slice 1.

## Adjacent items this feature leads into (named + scoped, NOT folded into Slices 0–2)
- **Cleanup A — remove the dead `_cgBoot` LOG-regeneration scaffolding.** `replaceWholeLogFromBuild(
  _domReadBuild())` is obsolete Phase-2-Step-3 seed code (its own comment says so; Step 5 made LOG
  authoritative). It defeats verbatim-LOG restoration on autosave and yields non-deterministic LOG shapes.
  Harmless today (compute is authoritative) but the natural ride-along cleanup since this work touches
  CharGen boot. **Its own change, its own test** — it currently "launders" the LOG and something may
  quietly depend on that; do not fold it silently into the switch diff.
- **Follow-on B — wire `creationLocked` into the CharGen→Live Sheet switch.** Phase 2's engine has a fully
  built "finalize creation" lock mechanism that **no UI triggers today** (verified: zero emission sites).
  "I'm done building, take me to the sheet to play" *is* that moment, so the CharGen→Live Sheet button is
  its natural home. Real rules/UX decision, explicitly **deferred** — named here because this feature
  enables it, not bundled into it.

## Files involved
- `js/character-store.js` **(new)** — the shared transport/storage/validation module described above.
- `tools/PACT-CharGen-Webtool.html` — import the module in the engine bridge; drop local `genCharId`; add
  `switchToLiveSheet()` + button; add handoff-consume (via existing `_cgApplyEnvelope`) + `sweepExpiredHandoffs`
  to `_cgBoot()` ahead of `loadFromHash`/autosave.
- `tools/PACT-Live-Char-Sheet.html` — import the module in the engine bridge; drop local `genCharId`; add
  `switchToCharGen()` + button; add handoff-consume (mirroring `importJSON`) + sweep to `_lsBoot()` ahead of `load()`.
- No changes to `js/engine.js`, `tools/DM-Console.html`, or any SQL/Supabase files.

## Out of scope
- Physical merge of the two HTML files.
- Collapsing to a single shared *character-save* slot (both tools reading one stored character) — the true
  end-state of "one shared model", but a bigger change this handoff is a stepping stone toward, not it.
- Any change to cloud sync / campaign binding / Supabase (verified against in Slice 2, not modified).
- Multi-character local storage / a character picker.
- File-based export/import (kept alongside the new button).
- Cleanup A and Follow-on B above (adjacent, separately scoped).
- Any `js/engine.js` / `compute()` / `DATA` rules change.

## Alternatives considered
- **Full one-file merge with a mode toggle** — reverses Phase 2's explicit decision; merging two huge files
  risks handler-name collisions for no gain over a redirect. Rejected by task owner.
- **Hand-copy the handoff glue into each tool (skip the shared module)** — the "obvious" cheaper path, but
  it would create a *third* copy of character-serialization logic and deepen the exact `js/`-vs-`tools/`
  violation the repo rule forbids; the module makes each tool's diff smaller, not larger. Rejected.
- **Encode the LOG into the URL** — URL length limits (~2000 chars) a big LOG can exceed; `localStorage` has
  no such limit. Rejected.
- **`sessionStorage`** — more naturally ephemeral, but this is a phone-first/installable PWA where
  navigation may not stay in one browsing context; `sessionStorage` would silently drop the handoff there.
  `localStorage` + per-transfer key + expiry + boot sweep gets the ephemerality without the fragility.
- **Single fixed handoff key** — original draft; a real two-tab race (simultaneous switches collide, last
  write wins). Per-transfer UUID keys fix it (at the cost of the orphan-sweep, now included).
- **`BroadcastChannel`/postMessage** — overkill for a same-tab navigation.

## Risks / open questions
- **Cloud/campaign-bound characters** — the biggest unknown; isolated to Slice 2 with a button-disable
  fallback so it can never block the local feature.
- **Browser back button** after a switch: query stripped via `replaceState`, so back should land on the
  source's prior state without re-triggering a stale handoff — needs a manual check.
- **Expiry window value** — plan uses ~2 minutes; sanity-check it survives a slow mobile navigation yet
  stays short enough to keep the staleness/orphan surface small.
- **Slice 0 module-bridge timing** — low risk (verified the engine-ready gating makes the module available
  before boot), but the one thing to smoke-test first: a shared function called from a classic script must
  resolve, i.e. the bridge must expose it on `window` before boot, exactly as it does for engine symbols.

## Verification
- `testing/tests/engine-parity.html` → still passing (no engine change).
- **Slice 0 (no-behavior-change):** both tools boot/save/load/generate-ids identically to before; a saved
  character reloads unchanged; `genCharId` still produces the same shape.
- **Happy path both directions:** build a Dwarf (racial trait + a drawback) in CharGen → "Open in Live
  Sheet" → identical character, no console errors → buy something → "Open in CharGen" → change reflected.
- **id stability:** identical `id` across CharGen → Live Sheet → CharGen → Live Sheet. (Byte-identical LOG
  is explicitly *not* required through a CharGen leg; compute-equivalence of the character is.)
- **Normal boot unaffected:** a plain reload (no `?handoff=`) still boots via existing autosave/last-saved.
- **Failure paths (all fall through to normal boot, no crash, no garbage):** corrupt JSON; `LOG` not an
  array; wrong/missing `schema`; expired `created`; the URL's uuid key absent.
- **Orphan sweep:** seed two extra expired `pact:handoff:*` keys, boot either tool, confirm both gone.
- **Rules-version banner:** hand off a character stamped with an older `rules`; destination shows its
  existing stale-rules/reprice indicator, not "current".
- **Reload doesn't re-trigger:** after consuming once, reloading the stripped URL consumes nothing.
- **Overwrite safety:** switching into a tool that already has a *different* autosaved character loads the
  new one and updates the slot only on success.
- **Browser back button:** lands on a sane prior state (manual).
- **Cloud/campaign (Slice 2):** same `id` before/after; the post-return cloud save updates the **same**
  remote record (not a new one); campaign membership/rules still visible; no duplicate in the DM roster.

## Done when
- `js/character-store.js` exists and both tools import it; the local `genCharId` duplicates are gone;
  Slice 0 shows no behavior change.
- Each tool has a clearly-labeled button that navigates directly to the other with the current character
  loaded — no file download/import — built on the shared module.
- Boot precedence is explicit; exactly one of {handoff, deep-link, autosave, blank} fires per boot.
- The consumed handoff key is always deleted; expired/orphan keys are swept on boot; an invalid handoff
  never breaks normal boot; the rules-version indicator fires on an older-rules handoff.
- All failure-path, id-stability, orphan-sweep, and overwrite-safety checks pass.
- Slice 2's cloud checks pass, **or** a decision is logged (`DECISIONS.md`) scoping the button to local-only
  characters for now.
- (Cleanup A and Follow-on B are explicitly not required for this plan's completion.)
